Problems with DevOps

DevOps solves a lot of problems, but presents some new ones variety: 20 teams means 20 different ci pipes, cd pipes, build systems, etc duplication: 1 or 2 ppl on every team solving the same problems commodities: 90% of the time product delivery devs just want to write code and have it run
  • Too much variety
  • Duplication of effort
  • Lack of commodities

Platform Engineering

Platform engineering picks up where devops leaves off Centralize expertise in CI/CD, and builds a self-service product around it. Teams still own their CI/CD, but the heavy lifting happens under the hood of the platform tooling This would be a "devops team". Once we have done that, we can expand the scope beyond a "devops team"
  • Centralize Expertise
  • Expand Scope

Platform Engineering

Infrastructure includes IaaS platform, such as CI/CD/k8s platform. this is language-agnostic, docker is the interface but usually (us and netflix, etc) it is more than that as well the service platform encompass language-specific tooling to improve the e2e developer experience, build tooling: eg bazel, gradle plugins platform services: kafka or other message buses. API gateway, discovery, cloud config server Runtime libraries: Spring boot extensions, libraries tp plugin into architecture (observability, platform services, etc)

Technical Philosophy

These principles led us to make a few high level tech decisions that provide some context on why we have implemented platform features the way we have

Technical Philosophy

Don't break the build

don't break downstream team's builds or other processes This doesn't mean we never change things It means we design for these changes How we do this will be covered more as we go
  • Don't disrupt productivity
  • Anticipate breaking changes
  • Help teams migrate
  • Track and report progress

Technical Philosophy

Monorepo vs Polyrepo

Core problem: How do we deliver shared build tooling?

Technical Philosophy

Cost of Monorepo

Build team: monorepo requires a highly customized or low-level build system which requires a dedicated team to build and maintain Scaling issues: Clone time / disk space. github codeowners We believe that we can solve the problem of managing changes and upgrades across our software components without taking on the downsides of a monorepo. It requires a little more work, but once it is in place, it gives us the best of both worlds, without introducing tight coupling at the repo layer or sacrificing autonomy.
  • Requires dedicated build team
  • Large git repos have scaling limitations
  • Large git repos have scaling limitations
  • Reduces Autonomy
  • Increases Coupling

Principles of Polyrepo

  • >Deliver shared tooling by plugging into CI and build systems
