Where we started: DevOps

DevOps solves a lot of problems, but presents some new ones variety: 20 teams means 20 different ci pipes, cd pipes, build systems, etc duplication: 1 or 2 ppl on every team solving the same problems commodities: 90% of the time product-based devs just want to write code and have it run

  • Too much variety
    -> difficult to maintain

  • Duplicate Solutions
    -> wasted effort across teams

  • Siloed teams
    -> lack of commodities slows down product development

What about Platform Engineering?

Platform engineering picks up where devops leaves off Centralize expertise in CI/CD and infrastructure management, and builds a self-service product around it. Teams still own their CI/CD, but the heavy lifting happens under the hood of the platform tooling This would be a "devops team". Once we have done that, we can expand the scope beyond a "devops team", and influence every aspect of the engineering lifecycle, in a streamlined way.

  • Centralize Devops Expertise
  • Expand Scope past operations into streamlined runtime solutions

Technical Philosophy

These principles led us to make a few high level tech decisions that provide some context on why we have implemented platform features the way we have

Technical Philosophy

Don't break the build

don't break downstream team's builds or other processes This doesn't mean we never change things It means we design for these changes How we do this will be covered more as we go
  • Don't disrupt productivity
  • Anticipate breaking changes
  • Help teams migrate
  • Track and report progress

Technical Philosophy

Monorepo vs Polyrepo

Core problem: How do we deliver shared build tooling?

Technical Philosophy

Cost of Monorepo

Build team: monorepo requires a highly customized or low-level build system which requires a dedicated team to build and maintain Scaling issues: Clone time / disk space. github codeowners We believe that we can solve the problem of managing changes and upgrades across our software components without taking on the downsides of a monorepo. It requires a little more work, but once it is in place, it gives us the best of both worlds, without introducing tight coupling at the repo layer or sacrificing autonomy.
  • Requires dedicated build team
  • Large git repos have scaling limitations
  • Reduces Autonomy
  • Increases Coupling

Technical Philosophy

Principles of Polyrepo

  • Keep related work together (iaas definitions, docs, etc)
  • Deliver shared tooling by plugging into CI and build systems

OpenRewrite

you may have noticed "automated migration" links using OpenRewrite, we were able to create a single gradle task a dev could run to automatically convert all mockito <4 code to mockito 4 compliant recipes are available for each spring upgrade, jdk version upgrades, and jakarta namespace migration we have also written custom recipes to migrate usage of our own deprecated roux code.

Evolution without disruption

using these tools, usage of @Deprecated in Roux went from 600 to 200 within about 9 months we were able to remove that old code, without breaking any builds in the process.
